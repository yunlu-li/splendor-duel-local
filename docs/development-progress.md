# 开发进度

本文记录当前代码状态，避免早期原型说明和实际实现脱节。

## 已完成

- Vite + React + TypeScript 项目骨架。
- 67 张珠宝卡 + 4 张皇室卡结构化本地数据：`src/shared/constants/cards.local.ts`。
- 5x5 中央棋盘建模与中心螺旋填充顺序：`src/shared/constants/board.ts`。
- seeded random 与可复现初始化：`src/shared/rules/initialize.ts`。
- 规则校验、reducer、计分和胜利判定：
  - 花特权拿非黄金 token。
  - 补充棋盘。
  - 强制拿 1-3 个非黄金 token，要求相邻、连续、同行/列/对角线。
  - 保留明牌或盲保牌堆，并拿黄金。
  - 购买市场卡或自己的保留卡，支持 bonus 折扣与黄金补缺口。
  - 弃 token 到 10 个。
  - 皇冠触发皇室卡选择。
  - 卡牌能力：额外回合、获得特权、拿同色 token、偷 token、Associate 复制 bonus。
- 可交互前端页面：
  - 本地双人模式。
  - 机器人练习模式。
  - 好友房入口、房间准备、邀请链接。
  - 棋盘选择、市场购买/保留、盲保、保留区购买。
  - 右侧卡牌详情与诏令日志。
  - 对手保留卡隐藏，只展示保留数量。
  - 资源不足时展示缺口。
- 服务端房间系统：
  - Fastify + Socket.IO 服务入口：`src/server/index.ts`。
  - 房间创建、加入、准备、离开与 action 广播：`src/server/rooms/RoomManager.ts`。
  - 联机模式下由服务端执行权威 reducer。
- 抽象卡牌 UI，不复刻官方图片。
- 测试：2 个测试文件，11 个测试用例。

## 当前限制

- 好友房已能跑通基础流程，但还没有断线重连、观战、房间持久化。
- 机器人能自动行动，但策略仍偏规则可行性，不是强策略 AI。
- 支付面板已支持手动选择支付 token，但交互仍可继续优化，例如更明确的黄金分配提示。
- 当前数据和卡牌命名用于本地原型与规则验证；公开发布时应继续避免使用官方素材和完整官方表达。

## 下一步建议

1. 给好友房增加断线重连和房间过期清理。
2. 给购买支付补充更完整的 UI 测试或浏览器流程测试。
3. 改进机器人策略，优先考虑可购买卡、皇冠、阻断对手资源。
